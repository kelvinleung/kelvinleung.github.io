JavaScript 的“编译”过程都做了什么？

OCT 29, 2021

一切要从网上看到的一道面试题说起：

code
1// 请写出下面代码的输出结果：2function fn(a, c) {3  console.log(a); // f a() {}4  var a = 123;5  console.log(a); // 1236  console.log(c); // f c() {}7  function a() {}8  if (false) {9    var d = 678;10  }11  console.log(d); // undefined12  console.log(b); // undefined13  var b = function () {};14  console.log(b); // f () {}15  function c() {}16  console.log(c); // f c() {}17}18fn(1, 2);

当时我瞄了一眼，这不很简单么，啪一下就做完了。回头一看答案，第一个console.log(a)居然不是1，什么情况？经过一番研究，查了无数资料，我终于搞明白了。原来 JS 居然也有“编译”过程，一些奇奇怪怪的事情都发生在这个阶段中。

可是，JS 明明是一门解析型语言，哪来的编译？

其实，这里所谓的“编译”，并不是“编译型语言”的那个编译，而是指在 JS 代码执行之前，JS 引擎所进行的一系列动作，像词法分析、变量初始化等。由于它们发生在代码真正“执行”之前，所以大家习惯称之为“编译”或“预编译”。了解 JS “预编译”的过程，对于理解 JS 的底层执行机制，如变量提升、this、闭包等，都会有很大的帮助。

执行上下文（Execution Context，下面简称 EC）

An execution context is a specification device that is used to track the runtime evaluation of code by an ECMAScript implementation.

所谓的 EC，指的是 JS 解析运行的环境，是一个抽象的概念。所有 JS 的代码，都是在 EC 中运行的。EC 有三种类型：

  • 全局执行上下文（Global Execution Context）：默认的 EC，所有不是写在函数里的代码，都运行在 GEC 中。GEC 会创建一个全局对象window（浏览器环境），并将 this 指向 window。所有 JS 程序有且只有唯一的一个 GEC。
  • 函数执行上下文（Function Execution Context）：每一次函数被调用时，都会为这个函数创建一个全新的 EC。每个函数都有自己的 EC，当且仅当在被调用时才会创建，而每个程序可以有无数个函数执行上下文。
  • Eval 函数执行上下文（Eval Function Execution Context）：每段在eval中执行的代码，都会有它对应的执行上下文。由于eval尽量还是少用，可以暂时忽略它。

每个 EC 都会有一系列的属性（或者叫状态）来记录与之关联的代码的执行进度，下面是 EC 的结构：

code
1ExecutionContext {2  VariableObject,3  ScopeChain,4  thisValue5}

除了上面三个属性以外，在不同的实现里还会有其它额外的状态。在详细说明上述三个属性前，我们先来了解一下什么是执行栈。

执行栈/调用栈（Execution Stack，下面简称 ES）

执行上下文的创建、搁置、恢复、结束等，是通过执行栈来管理的。执行栈（有些语言也叫“调用栈”，Calling Stack）顾名思义，是一个后进先出的“栈”，用于存放所有 JS 代码执行过种中产生的执行上下文。JS 引擎在开始执行代码之前，首先会创建一个 GEC，然后把它压入执行栈中。当代码执行到有函数调用的时候，又会为该函数创建 EC 并压进 ES。之前的 EC 则会保存当前的执行状态，进入到搁置状态中。JS 引擎接着执行刚压进栈顶的 EC 所对应的函数，当该函数执行完毕后，它的 EC 从 ES 中弹出，然后将之前搁置的 EC 恢复，继续执行其对应的函数，直至所有代码执行完毕。执行完毕后，最底层的 GEC 最后弹出，整个程序执行完毕。以下面的代码为例：

code
1// 1. 先创建一个 GEC，入栈2let a = "Hello World!";3function first() {4  console.log("Inside first function");5  second(); // 3. 调用 second() 时创建一个 FEC，入栈，second-EC6  // 4. 执行 second() 结束，弹出 second-EC7  console.log("Again inside first function");8}9function second() {10  console.log("Inside second function");11}12first(); // 2. 调用 first() 时创建一个 FEC，入栈，first-EC13// 5. 执行 first() 结束，弹出 first-EC14console.log("Inside Global Execution Context");15// 6. 代码执行完毕，弹出 GEC

程序的执行过程为：

  1. 创建一个全局执行上下文，压入执行栈
  2. 执行全局代码，到第 12 行，调用 first() 函数，为其创建函数执行上下文，first-FEC，压入执行栈
  3. 执行 first() 中的代码，到第 5 行，调用 second() 函数，为其创建函数执行上下文，second-FEC，压入执行栈
  4. 执行 second() 中的代码，结束后弹出 second-FEC
  5. 继续执行 first() 中的代码，结束后弹出 first-FEC
  6. 继续执行全局代码，结束后弹出 GEC，整个程序结束运行

VO（Variable Object，变量对象）

“预编译”阶段，一个很重要的任务是初始化变量。我们定义的变量，声明的函数，函数的形参等用于执行代码的数据，都存放在与对应执行上下文关联的一个叫变量对象（VO）的机制中。VO 中存储着在上下文中声明的以下内容：

  • 变量（var，VariableDeclaration）
  • 函数声明（FunctionDeclaration)
  • 函数的形参

可以表示为以下的形式：

code
1// 活跃的执行上下文2activeExecutionContext = {3  VO: {4    // 上下文数据（变量，函数声明，函数形参）5  },6};

当我们定义变量或声明函数时，相当于给 VO 创建新的属性，比如下面的代码：

code
1var a = 10;2function test(x) {3  var b = 20;4}5
6test(30);

相对应的 VO 可以表示为：

code
1// 全局上下文的 VO2VO(globalContext) = {3  a: 10,4  test: <reference to function> // 指向函数5};6
7// test() 函数上下文的 VO8VO(test functionContext) = {9  x: 30,10  b: 2011};

VO 可以看作是一个抽象的概念，它在不同的执行上下文中，有着不同的命名与初始结构。

  • 在全局执行上下文中，VO 即全局对象window（浏览器环境），又被称为 GO（Global Object）；它在初始化的时候，定义了像Math、String、Date等属性
  • 在函数执行上下文中，它还附加arguments、形参等额外的细节，又被称为 AO（Activation Object）

了解了 VO 的概念后，就可以正式进入主题了。在旧规范中，执行上下文的创建分为两个基本阶段：

  • 进入执行上下文
  • 执行代码

阶段一：进入执行上下文

在进入执行上下文时，VO 会添加以下属性：

  1. 对于每一个形参（假设是在函数执行上下文中，此时的 VO 称为 AO），会增加一个名称和值与该形参对应的属性；如果是未传值的形参，则值为undefined
  2. 对于每个函数声明，会增加一个名称与函数名一样的属性，值为该函数对象；如果 VO 已经有同名属性，则覆盖之
  3. 对于每个变量声明，会增加一个名称与该变量一样的属性，值为undefined；如果 VO 已经有同名属性，则保留原有的属性，忽略当前变量声明

下面用一个例子来展示上述过程：

code
1function test(a, b) {2  var c = 10;3  function d() {}4  var e = function _e() {};5  (function x() {});6}7
8test(10); // 调用函数

在调用函数 test() 时，传进去了参数10，此时在进入执行上下文阶段中，AO 可以表示为：

code
1AO(test) = {2  a: 10,3  b: undefined,4  c: undefined,5  d: <reference to FunctionDeclaration "d"> // 指向函数 d6  e: undefined7};

有两个个地方要注意一下：

  • 首先，d的值是一个函数，因为它是一个函数声明，而e的值是undefined，因为它是一个函数表达式，e是一个变量
  • x并没有在 AO 里，因为它也是一个表达式，但并没有赋值给任何变量，不存在变量的声明

阶段二：执行代码

接着上面的例子，当进入执行代码阶段的时候，test()函数中的代码开始执行，AO 中的属性被赋值：

code
1AO['c'] = 10;2AO['e'] = <reference to FunctionExpression "_e">;

此时，AO 可以表示为：

code
1AO(test) = {2  a: 10,3  b: undefined,4  c: 10,5  d: <reference to FunctionDeclaration "d"> // 指向函数 d6  e: <reference to FunctionExpression "_e"> // 指向函数 _e7};

值得注意的是，e的值为_e函数体，但该函数是不可以通过_e()来调用的，因为它并没有在 AO 的声明。我们再通过一个例子来加深对赋值表达式以及这两个阶段的理解：

code
1console.log(x); // function x() {}2var x = 10;3console.log(x); // 104x = 20;5function x() {}6console.log(x); // 20

类似var x = 10这样的赋值表达式，可以拆解成两个独立的语句来理解。首先是var x，定义了一个变量x；然后是x = 10，给这个变量赋上值。我们将上述示例代码的第 2 行，拆成这样的两段，整个执行上下文的创建过程，可以这样分析：

  1. 首先是第一阶段，代码执行到第 2 行的前半部分，也就是var x，此时VO = { x: undefined }
  2. 接着代码执行到第 5 行，是一个函数的声明，尽管 VO 中已经有了同名的x属性，但由于函数声明会将其覆盖，此时VO = { x: <reference to FunctionDeclaration "d"> }
  3. 第一阶段结束，进入第二阶段
  4. 代码执行到第 1 行，此时x为函数，打印出函数体
  5. 代码跳过第 2 行的前半部分（已经在第一阶段执行了），执行后半部分，x = 10，此时VO = { x: 10 }
  6. 代码执行到第 3 行，打印x的值10
  7. 代码执行到第 4 行，此时VO = { x: 20 }
  8. 代码跳过第 5 行（已经在第一阶段执行了）
  9. 代码执行到第 6 行，打印x的值20

如果你能理解这个过程，那文章开头的那道题，也可以用同样的思路去理解。

作用域链（Scope Chain）

所谓的作用域链，是一连串 VO 所组成的“链”，它决定了某个执行上下文中，能够访问到哪些变量及函数。例如：

code
1var a = 10;2function foo() {3  var b = 12;4  var c = addANumber(20);5  function addANumber(num) {6    return a + num;7  }8  console.log(c); // 309}10foo();

在上面的例子中，存在三个执行上下文：

  • 全局执行上下文，其 GO 包含变量a与函数foo()
  • foo()的函数执行上下文，其 AO 包含变量b、c与函数addNumber()
  • addNumber()的函数执行上下文，其 AO 包含形参num

其中，函数addNumber()可以访问到其 AO 上的形参num以及其外层上下文 VO 中的所有属性。比如在addNumber()中访问变量a，首先会查看当前 AO，发现当前作用域中没有该变量，于是往外层继续查找，在foo()的 AO 中，同样找不到变量a，再继续往外来到全局上下文，在 GO 中找到了a。通过这样的方式，形成的查找链，就是作用域链。像变量a这种不在当前作用域中的变量，我们称之为“自由变量”，查找自由变量的过程，就需要用到作用域链。

要注意的是，内层能访问外层，而外层没法访问内层。你无法在全局作用域中调用addNumber()函数，因为它定义在内层的 VO 中。

“this”

this是执行上下文中的一个特殊对象，任何对象都有可能成为this的值。this的值在进入执行上下文的时候就被确定，且无法被修改，因为它并不是一个位于 VO 上定义的“变量”。要注意的是，在 ES6 之前，this是执行上下文的属性，而不是 VO 的属性；而在 ES6 中，this成为了词法环境的属性，目的是为了支持箭头函数（箭头函数的this继承自其外层执行上下文）。

在全局上下文中，this的值等于全局对象 GO：

code
1var x = 10;2console.log(3  x, // 104  this.x, // 105  window.x // 106);

在函数上下文中，this的值在每一次函数调用中都有可能不同，this的值通过调用表达式（调用方式）由调用方（caller）提供。下面的例子中，函数foo()的代码自始至终并没有改变过，但this的值（caller）却在每次调用中都不一样：

code
1function foo() {2  alert(this);3}4
5foo(); // caller: 全局对象6foo.prototype.constructor(); // caller: foo.prototype7
8var bar = {9  baz: foo,10};11
12bar.baz(); // caller: bar13
14bar.baz(); // caller: bar15(bar.baz = bar.baz)(); // caller: 全局对象16(bar.baz, bar.baz)(); // caller: 全局对象17(false || bar.baz)(); // caller: 全局对象18
19var otherFoo = bar.baz;20otherFoo(); // caller: 全局对象

新规范（ES5+）中的 EC

在我一开始去找有关变量提升题目，或者说“预编译”的解释资料时，我发现不同版本的规范对这个过程有着不同的定义。一开始我还以为是我理解错了，很多概念、过程都对不上。后来我又查了更多资料，还看了 ES 的规范，才发现是新的规范里作了调整，不过基本上是大同小异。在网上能找到的资料中，尤其是国内的中文语境下，大多还是套用上文提到的 VO 的执行过程，基于 ES1/3，早已过时。

而在 ES5 及以后的版本中，已经不存在 VO 等一系列相关的概念了，取而代之的是一个叫词法环境（Lexical Environment）的新定义。ES6 标准中对词法环境的定义为：

A Lexical Environment is a specification type used to define the association of Identifiers to specific variables and functions based upon the lexical nesting structure of ECMAScript code. A Lexical Environment consists of an Environment Record and a possibly null reference to an outer Lexical Environment.

简单来说，词法环境就是一个包含了标识符与变量间映射关系的结构（这里的标识符指的是变量/函数名，变量指的是实际的对象以及原始值，其中对象又包括函数对象与数组对象）。以下面的代码为例：

code
1function foo() {2  var bar = 10;3}

变量bar在“词法”上位于foo()函数内部，而foo()函数在词法上又位于全局对象中，这些词法结构被记录在词法环境中。执行上下文创建的时包括以下两个组件（component）：

  • 词法环境（LexicalEnvironment）
  • 变量环境（VariableEnvironment)

它们都属于词法环境（Lexical Environment，上面的 LexicalEnvironment 可以看作是它的实例）。因此，执行上下文也可以用以下的伪代码表示：

code
1ExecutionContext = {2  LexicalEnvironment, // 指向内存中的词法环境3  VariableEnvironment, // 指向内存中的变量环境4};

它们的区别在于，LexicalEnvironment用于存储函数声明和let、const声明的变量，而VariableEnvironment仅用于存储var变量。

词法环境（Lexical Environment），包含了以下三个组件：

  • 环境记录（Environment Record）
  • 外部环境引用（Reference to the Outer Environment）
  • This 绑定（This Binding）

环境记录（Environment Record）

环境记录是词法环境中用来记录变量与函数声明的地方，它又分为两类：

  • 声明性环境记录（Declarative Environment Record）：存储变量和函数声明，对于函数来说，还包括了arguments对象
  • 对象环境记录（Object Environment Record）：全局代码的词法环境，除了全局变量与函数声明外，还包括了由宿主环境提供的对象属性（以浏览器为例，即提供给window对象的各种属性）

这些概念听上去是不是很熟悉，其实本质上就是 VO（AO/GO），变量创建赋值的机制也是一样的，分为两个阶段，这里不再赘述。

外部环境引用（Reference to the Outer Environment）

对于外层词法环境的引用，JS 引擎在对变量进行查找时，若不能在当前词法环境中找到该变量，则会利用这个引用，从外层的词法环境中继续查找。说白了，就是上面提到的作用域链。

This 绑定（This Binding）

顾名思义，就是用来记录this的地方，它在 ES5 的时候还是执行上下文的属性，而在 ES6 时又改为词法环境中的一个组件。

小结

最后，我们用一个例子来把这些新概念串起来：

code
1let a = 20;2const b = 30;3var c;4function multiply(e, f) {5  var g = 20;6  return e * f * g;7}8c = multiply(20, 30);

上述代码执行的时候，JS 引擎首先会创建一个全局的执行上下文来执行全局代码。在执行上下文的创建阶段时，全局上下文可以表示为：

code
1GlobalExectionContext = {2  LexicalEnvironment: {3    EnvironmentRecord: {4      Type: "Object",5      // 标识符绑定6      a: < uninitialized >, // 未初始化7      b: < uninitialized >,8      multiply: < func >,9    },10    outer: <null>,11    ThisBinding: <Global Object>,12  },13  VariableEnvironment: {14    EnvironmentRecord: {15      Type: "Object",16      // 标识符绑定17      c: undefined,18    },19    outer: <null>,20    ThisBinding: <Global Object>,21  }22}

接下来进入到全局上下文的执行阶段，变量赋值在此完成。此时的全局上下文为：

code
1GlobalExectionContext = {2  LexicalEnvironment: {3    EnvironmentRecord: {4      Type: "Object",5      a: 20,6      b: 30,7      multiply: < func >,8    },9    outer: <null>,10    ThisBinding: <Global Object>,11  },12  VariableEnvironment: {13    EnvironmentRecord: {14      Type: "Object",15      c: undefined,16    },17    outer: <null>,18    ThisBinding: <Global Object>,19  }20}

程序继续执行到multiply(20, 30)时，一个新的函数执行上下文被创建来执行multiply()函数。此时函数上下文进入到创建阶段：

code
1FunctionExectionContext = {2  LexicalEnvironment: {3    EnvironmentRecord: {4      Type: "Declarative",5      Arguments: {0: 20, 1: 30, length: 2},6    },7    outer: <GlobalLexicalEnvironment>,8    ThisBinding: <Global Object or undefined>, // 严格模式下为 undefined9  },10  VariableEnvironment: {11    EnvironmentRecord: {12      Type: "Declarative",13      g: undefined,14    },15    outer: <GlobalLexicalEnvironment>,16    ThisBinding: <Global Object or undefined>,17  }18}

随后函数上下文进入到执行阶段，为变量进行赋值操作，此时上下文为：

code
1FunctionExectionContext = {2  LexicalEnvironment: {3    EnvironmentRecord: {4      Type: "Declarative",5      Arguments: {0: 20, 1: 30, length: 2},6    },7    outer: <GlobalLexicalEnvironment>,8    ThisBinding: <Global Object or undefined>,9  },10  VariableEnvironment: {11    EnvironmentRecord: {12      Type: "Declarative",13      g: 2014    },15    outer: <GlobalLexicalEnvironment>,16    ThisBinding: <Global Object or undefined>,17  }18}

最后，当函数执行完毕后，会接返回值存储在变量c中，全局上下文的词法环境得到更新，当所有全局代码完成执行后，整个程序执行结束。

你可能注意到了，由let和const声明的变量，并没有初始值，而由var声明的变量的初始值为undefined。因此你可以在var声明的变量之前使用该变量（尽量值为undefined），却不能在let或const声明的变量前使用它们（会报错）。而这，就是所谓的“变量提升”。

在执行阶段时，如果 JS 引擎没法找到let声明变量的赋值时，该变量则会被赋值为undefined。
